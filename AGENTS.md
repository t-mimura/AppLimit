# AGENTS.md — AIエージェント向けガイド

このプロジェクトはAIエージェントが全工程を開発する。
このドキュメントはエージェントが作業する際の行動指針を定める。

## 基本原則

1. **仕様に従う** — [docs/SPEC.md](docs/SPEC.md) が唯一の仕様源。仕様にない機能を追加しない。
2. **テストファースト** — 実装コードより先にテストを書く。詳細は [docs/TEST_POLICY.md](docs/TEST_POLICY.md)。
3. **コーディングルールを守る** — [docs/CODING_RULES.md](docs/CODING_RULES.md) に従う。
4. **小さく作る** — 1つのコミットは1つの論理的変更に対応させる。
5. **動く状態を保つ** — コミットごとに `./gradlew check` が通る状態を維持する。

## 作業フロー

### コードを書くとき

```
1. テストを書く（Red）
2. テストが失敗することを確認
3. 実装コードを書く（Green）
4. テストが通ることを確認
5. リファクタリング（Refactor）
6. ./gradlew ktlintFormat を実行
7. ./gradlew check が通ることを確認
8. コミット
```

### コミットメッセージ

```
<type>: <簡潔な説明>

<必要に応じて詳細>
```

type は以下のいずれか:
- `feat` — 新機能
- `fix` — バグ修正
- `refactor` — リファクタリング
- `test` — テストの追加・修正
- `docs` — ドキュメント
- `chore` — ビルド設定、依存関係等

### ブランチ戦略

- `main` — 常にビルド可能な状態
- `feat/<機能名>` — 機能開発用ブランチ
- 機能完了後に main へマージ

## エラーが発生した場合

1. エラーメッセージを正確に読む
2. 原因を特定してから修正する（推測で修正しない）
3. 修正後、必ず `./gradlew check` で全体が壊れていないことを確認する

## ファイル作成のルール

- 新しいクラスを作る前に、既存のクラスに追加できないか検討する
- 1ファイルにつき1つの公開クラス/インターフェースを原則とする
- テストファイルは対応するソースファイルと同じパッケージ構成にする
  - `app/src/main/.../data/repository/AppRepository.kt`
  - `app/src/test/.../data/repository/AppRepositoryTest.kt`

## 権限・API使用時の注意

- `UsageStatsManager` はシステム設定から手動で権限付与が必要（通常の権限リクエストダイアログでは付与できない）
- `SYSTEM_ALERT_WINDOW` は `Settings.ACTION_MANAGE_OVERLAY_PERMISSION` で設定画面に誘導する
- これらの権限が未付与の場合、アプリ起動時にわかりやすく誘導UIを表示すること

## サブエージェントへの指示

サブエージェントにタスクを委譲する場合、以下を必ず伝達すること:

1. このファイル（AGENTS.md）のパスと読むべき指示
2. 対象タスクの具体的なスコープ（どのファイルを作成/変更するか）
3. 完了条件（何をもって完了とするか）
4. `./gradlew check` を実行して通ることの確認指示
