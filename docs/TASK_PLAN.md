# タスク分割・実行計画

## フェーズ構成

開発を5つのフェーズに分割する。各フェーズは前のフェーズの成果物に依存するため、順番に実行する。

---

## Phase 0: 環境構築

**目的**: ビルド・テスト・品質チェックが動く空のプロジェクトを用意する

| # | タスク | 実行方法 | 完了条件 |
|---|--------|---------|---------|
| 0-1 | JDK 17 / Android SDK インストール | メインエージェント | `java --version` が動く、`sdkmanager --list` が動く |
| 0-2 | Gradleプロジェクト生成 | メインエージェント | `./gradlew assembleDebug` が成功する |
| 0-3 | ktlint 導入 | メインエージェント | `./gradlew ktlintCheck` が動く |
| 0-4 | Hilt / Room / Navigation 等の依存関係追加 | メインエージェント | ビルドが通る |
| 0-5 | pre-commit フック設定 | メインエージェント | フック経由で ktlint + test が実行される |
| 0-6 | Git リポジトリ初期化・初回コミット | メインエージェント | `git log` でコミットが確認できる |

**Phase 0 はすべてメインエージェントが順次実行する。** 環境構築は依存関係が多く、並列化のメリットが薄いため。

---

## Phase 1: データ層

**目的**: 設定データの保存・読み出しの基盤を作る

| # | タスク | 実行方法 | 完了条件 |
|---|--------|---------|---------|
| 1-1 | ドメインモデル定義 (`TargetApp`, `SessionState` 等) | サブエージェント | モデルクラスが存在し、単体テストが通る |
| 1-2 | Room Entity / Dao / Database 定義 | サブエージェント | Robolectric テストで CRUD 操作が通る |
| 1-3 | Repository インターフェース + 実装 | サブエージェント | Fake を使った単体テストが通る |
| 1-4 | Hilt モジュール (データ層) | サブエージェント | DI でRepository が注入できることをテストで確認 |

**1-1 → 1-2 → 1-3 → 1-4 の順で実行。** 各タスクが前のタスクの成果物に依存するため。
1つのサブエージェントが Phase 1 全体を担当する。

---

## Phase 2: サービス層

**目的**: アプリ監視・タイマー・オーバーレイの中核ロジックを作る

| # | タスク | 実行方法 | 完了条件 |
|---|--------|---------|---------|
| 2-1 | UsageStats ラッパー (フォアグラウンドアプリ検出) | サブエージェント A | インターフェース定義 + Fake + テストが通る |
| 2-2 | タイマー管理ロジック (セッション・クールダウン状態遷移) | サブエージェント A | 全状態遷移のテストが通る |
| 2-3 | 監視フォアグラウンドサービス | サブエージェント A | サービスが起動・停止できる |
| 2-4 | オーバーレイ表示ロジック | サブエージェント B | オーバーレイの表示・非表示・延長がテストで確認できる |
| 2-5 | 通知ロジック (残り時間警告、制限発動) | サブエージェント B | 通知が発行されることをテストで確認 |
| 2-6 | BootReceiver (端末再起動時のサービス自動起動) | サブエージェント B | Receiver が登録されビルドが通る |

**並列実行可能:**
- サブエージェント A: 2-1 → 2-2 → 2-3 (監視・タイマー系)
- サブエージェント B: 2-4 → 2-5 → 2-6 (UI・通知系)

2-4 はオーバーレイの Compose UI ではなく、表示判定ロジックとWindowManager操作のラッパーを指す。

---

## Phase 3: UI層

**目的**: ユーザーが操作する画面を作る

| # | タスク | 実行方法 | 完了条件 |
|---|--------|---------|---------|
| 3-1 | メイン画面 (対象アプリ一覧) + ViewModel | サブエージェント C | ViewModel のテストが通る、Compose Preview が表示される |
| 3-2 | アプリ選択画面 + ViewModel | サブエージェント D | ViewModel のテストが通る、Compose Preview が表示される |
| 3-3 | アプリ個別設定画面 + ViewModel | サブエージェント D | ViewModel のテストが通る、Compose Preview が表示される |
| 3-4 | オーバーレイ画面 (Compose UI) | サブエージェント C | 表示要素が仕様通りであること |
| 3-5 | 権限リクエスト画面・誘導UI | サブエージェント C | 未付与時に誘導画面が表示される |
| 3-6 | Navigation 統合 (画面遷移) | メインエージェント | 全画面間の遷移が動作する |

**並列実行可能:**
- サブエージェント C: 3-1 → 3-4 → 3-5
- サブエージェント D: 3-2 → 3-3
- 3-6 は C, D 完了後にメインエージェントが実行

---

## Phase 4: 結合・仕上げ

**目的**: 全体を結合し、リリース可能な状態にする

| # | タスク | 実行方法 | 完了条件 |
|---|--------|---------|---------|
| 4-1 | 全層結合・動作確認 | メインエージェント | `./gradlew assembleDebug` が成功、全テスト通過 |
| 4-2 | AndroidManifest 最終整理 (権限、サービス、Receiver) | メインエージェント | ビルド・Lint 通過 |
| 4-3 | アプリアイコン・テーマ設定 | メインエージェント | リリース用のアイコンとテーマが設定されている |
| 4-4 | リリースビルド設定 (signing, ProGuard) | メインエージェント | `./gradlew assembleRelease` が成功 |
| 4-5 | 最終 `./gradlew check` 通過確認 | メインエージェント | 全チェック通過 |

**Phase 4 はメインエージェントが順次実行。** 結合作業は全体を把握して行う必要があるため。

| サブエージェント D | Phase 3 (後半) | アプリ選択・設定画面 |

## 並列実行の判断基準

- 同一ファイルを変更しない
- 依存方向が一方向（下位層が上位層に依存しない）
- インターフェースが事前に定義済み

これらを満たす場合に並列実行可能と判断する。
